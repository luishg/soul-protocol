The Idea

Identity + Memory + Context
should belong to you

Soul Protocol defines a portable container for an AI assistant. A "soul" that can live locally, be versioned, backed up, and re-attached to any compatible model.

The Structure

A minimal soul in four files

Human-readable. Plain text. Git-friendly. You should be able to read your AI's soul.

identity.md

Who the assistant is

Name, personality, tone, values, role, and boundaries. Stable over time — rarely changes.

user.md

Who the assistant is for

Preferences, communication style, language, expectations. Defines the bond, not the assistant alone.

memory.md

What the assistant remembers

Long-term facts, learned preferences, important events. Curated semantic memory — not infinite logs.

context.md

How the assistant thinks

System rules, reasoning preferences, tool philosophy, safety constraints. The operating philosophy of the soul.

Design Principles

Built on convictions

Portable

Works across local models, cloud models, and future models we don't know yet. No provider lock-in.

Human-readable

Markdown by default. Plain text. Git-friendly. Auditable. You should be able to read your AI's soul.

User-owned

Stored locally. Encrypted if synced. Never implicitly shared. Your AI remembers for you, not about you.

Model-agnostic

Defines no model, no UI, no runtime. Soul Protocol defines identity continuity — nothing more.

Clarity

A standard, not a product

Soul Protocol is not a chatbot framework, not a memory database, not an agent platform, not a prompt marketplace. It's lower-level: an open specification for identity continuity.
